### 2026-06-17 Web-First Static MVP

Status: Accepted
Topics: product, architecture, corpus, licensing
Refs: SPEC.md, README.md, ../idealog/specs/relevant-xkcd.md

Decision:
Build Relevant xkcd as a web-first search/share app. Start with a static
client-side implementation, using TypeScript for the app, ranking, tests, and
future corpus/index generation. Keep the existing Python scripts as disposable
corpus scaffolding unless they prove cheaper to retain temporarily.

Static search is the first baseline, not a permanent constraint. A search API,
offline embeddings, or a hosted vector database may be introduced later if
measured relevance or performance shows that static browser search is
insufficient.

Do not commit the full raw corpus by default. Keep `raw_data/` ignored, commit a
small deterministic fixture set for tests, and publish only the minimized search
index needed by the static app.

For the public MVP, prefer xkcd-provided metadata and links. Do not ship
explainxkcd-derived text in public assets until attribution, source links,
CC BY-SA handling, and generated-index licensing are explicit.

Rationale:
The core product risk is search quality and share speed, not platform
integration. A static web app has the lowest delivery friction and keeps the
first version usable in texts, work chats, and other conversations without
Apple-specific integration.

Search quality may eventually require stronger retrieval than a browser-side
lexical index. That should be a measured escalation because an API or vector
database adds hosting, uptime, deployment, and maintenance responsibilities.

The local corpus is useful prior work, but raw snapshots are source data rather
than hand-authored project code. Keeping the full corpus out of git avoids noisy
vendor-data churn and reduces accidental redistribution risk.

Alternatives:
- Start with an iMessage extension. Rejected for MVP because it increases
  platform friction before search quality is proven.
- Start with a hosted vector database. Deferred because it adds operational
  complexity before the static baseline has been measured.
- Track the full raw corpus in git. Deferred because it increases repo churn and
  license/attribution surface without being necessary for deterministic tests.
- Use explainxkcd text immediately in the public index. Deferred until license
  handling is implemented deliberately.

### 2026-06-19 Resumable Corpus Fetching

Status: Accepted
Topics: corpus, operations, licensing, search quality
Refs: README.md, scripts/fetch-corpus.ts, scripts/corpus-status.ts

Decision:
Manage corpus updates through foreground, resumable npm scripts rather than a
daemon or browser-triggered network path. Fetching is single-threaded by
default, skips existing raw files unless `--refresh` is passed, writes ignored
run logs and a manifest under `raw_data/`, and uses a lock file to prevent two
fetches from running at once.

Use a default delay of at least 1500 ms between requests to the same source.
Fetch xkcd metadata and explainxkcd pages into separate raw directories, then
build public search records from local files without requiring network access.

Allow explainxkcd-derived transcript/explanation text in the generated search
index only as provenance-aware fields with visible app attribution and source
links.

Rationale:
The project needs fresher and richer text to evaluate search quality, but both
source sites should be treated respectfully. A foreground resumable command is
easy to monitor, safe to interrupt, and avoids hidden background traffic.

Keeping raw source snapshots ignored preserves the original corpus policy while
making the generated index reproducible from local files. Separating xkcd and
explainxkcd fields keeps license and source attribution explicit.

Alternatives:
- Parallel fetching. Rejected for now because the corpus is small and source
  courtesy matters more than speed.
- A long-running fetch service. Rejected because corpus updates are occasional
  maintenance tasks, not application runtime behavior.
- Merging explainxkcd text into anonymous `searchText` only. Rejected because
  provenance and attribution need to remain visible.

### 2026-06-19 Mixed License Boundary

Status: Accepted
Topics: licensing, public repository, corpus
Refs: LICENSE.md, THIRD_PARTY_NOTICES.md, README.md

Decision:
License original project code under MIT while excluding third-party content and
generated data derived from third-party content from that grant. Document xkcd
content as Creative Commons Attribution-NonCommercial 2.5 and explainxkcd wiki
content as Creative Commons Attribution-ShareAlike 3.0 in a dedicated third
party notice file.

Keep public use noncommercial while generated assets include xkcd-derived
content. Preserve source provenance and attribution notices for generated
indexes and fixtures.

Rationale:
A standard software license is appropriate for the app code, scripts, tests, and
project documentation, but the repository also contains generated search assets
and fixtures derived from upstream licensed content. A blanket repository-wide
MIT signal would imply rights the project cannot grant.

Separating original code licensing from upstream content notices keeps the repo
open for software reuse while making the restrictions on redistributed corpus
data explicit before public release.

Alternatives:
- Apply MIT to the whole repository. Rejected because it would be misleading for
  generated indexes and fixtures that include xkcd and explainxkcd content.
- License everything under a Creative Commons license. Rejected because Creative
  Commons recommends software-oriented licenses for software code, and this
  would unnecessarily complicate code reuse.
- Remove all generated/indexed third-party content before publishing. Deferred
  because the current MVP needs a public search index, and the sources permit
  noncommercial redistribution with proper attribution and license handling.

### 2026-06-20 Explainxkcd Transcript Boundary

Status: Accepted
Topics: corpus, search quality, licensing
Refs: src/lib/normalize.ts, src/lib/search.ts, public/search-index.json

Decision:
Use explainxkcd-derived transcript text as supplemental comic-visible text.
When an xkcd transcript exists, score the community transcript at a lower
supplemental weight. When the xkcd transcript is missing, score the community
transcript at normal transcript weight. Do not index explainxkcd explanation
text, related comic links, or wiki categories for lexical or semantic search.

Rationale:
Community transcripts materially improve coverage and can add better visible
text even when an xkcd transcript exists, but they are noisier and should not
overpower first-party transcript text. Explanation prose and wiki references
broaden recall but add noisy matches and make the product feel like explainxkcd
search instead of xkcd search.

Alternatives:
- Use community transcripts only as fallback coverage. Rejected because it
  discards useful visible-text improvements for comics that also have xkcd
  transcripts.
- Index explanations and references at low weight. Rejected for MVP because the
  extra recall is less important than predictable, explainable search behavior.

## 2026-06-26 Grouped Index Releases

- Status: Accepted
- Topics: corpus, index, deployment, caching, automation
- Refs: SPEC.md, public/search-index.json, public/semantic-index.json
- Decision: Target a scheduled GitHub Actions updater for the deployed GitHub
  Pages app. Use the published generated index as updater baseline state, fetch
  new and recent source records, upsert generated records, rebuild semantic and
  lexical assets, and validate the assets as one compatible release.
- Rationale: The app needs same-day freshness without requiring a server or
  public raw corpus archive. Generated indexes are sufficient durable state for
  routine updates, while a manifest or equivalent build metadata can prevent
  lexical and semantic assets from drifting.
- Alternatives: Keep overwriting `/search-index.json` and
  `/semantic-index.json` without release metadata; rejected because automation
  needs compatibility checks. Require a complete raw snapshot in automation;
  rejected because raw snapshots are useful maintainer caches, not deployment
  state. Shard indexes immediately; deferred until measured bandwidth or mobile
  startup cost justifies the loader complexity.

## 2026-06-29 Checked-In Index Build Input

- Status: Accepted
- Topics: build, CI, corpus, index
- Refs: package.json, scripts/build-index.ts, README.md
- Decision: Normal dev, build, verify, CI, and deployment commands consume the
  checked-in public index assets. They do not regenerate public indexes from
  ignored `raw_data/` snapshots. Any raw snapshot conversion must opt in with an
  explicit source path.
- Rationale: `raw_data/` is a maintainer archive that may be stale, incomplete,
  or locally divergent. Treating it as an implicit build input causes accidental
  public index churn and can desynchronize search and semantic assets.
- Alternatives: Keep auto-generating from `raw_data/` when present; rejected
  because a local archive should not affect routine builds. Remove raw
  conversion tooling entirely; deferred because explicit conversion remains
  useful for one-off source debugging and experiments.

## 2026-06-29 One-Record-Per-Line Search Index

- Status: Accepted
- Topics: index, diffs, deployment
- Refs: public/search-index.json, scripts/lib/public-index.ts
- Decision: Format the public search index as a valid JSON array with one comic
  record per line, and make updater/generator scripts preserve that format.
- Rationale: The app can keep using `response.json()`, while generated index
  diffs become reviewable at record granularity. The size increase is about one
  newline per record and compresses well.
- Alternatives: Keep single-line JSON; rejected because any record change
  appears as a whole-file line replacement. Use JSONL; rejected for now because
  it would require custom loader/parsing behavior for little benefit.
